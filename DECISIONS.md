## Kasparro AI — Backend Engineering

The most foundational architectural decision I made was to implement an append-only transaction ledger rather than storing a mutable credit balance directly on the organization record. A traditional design would store an integer balance and update it whenever credits were added or deducted. While that approach is straightforward, it introduces significant risks in a concurrent, distributed system. Every update becomes a read–modify–write cycle, which is vulnerable to race conditions when multiple requests attempt to modify the same value simultaneously. In financial systems, even a single corrupted update is unacceptable. The append-only ledger eliminates that risk by treating every credit change as an immutable event. Instead of modifying a balance, the system records transactions. The current balance is derived from the sum of those transactions. This ensures that the source of truth is always the complete history of events rather than a fragile, continuously overwritten number.

I considered the performance implications of deriving the balance dynamically on every request. A mutable balance column would be computationally cheaper and simpler to query. However, that simplicity comes at the cost of auditability and correctness. In contrast, the ledger approach guarantees a mathematically perfect audit trail. If an organization disputes a charge, the system can reconstruct the exact account state at any point in time. This was a decisive factor. Financial correctness and audit transparency were prioritized over immediate performance optimization. I concluded that performance issues can be addressed later with caching or snapshotting strategies, but financial inconsistency is far harder to repair after the fact.

Handling concurrency in a multi-tenant API environment required careful thought. Even with an append-only ledger, the system must prevent organizations from exceeding their allocated credits when multiple requests arrive at nearly the same moment. I evaluated whether to rely purely on database-level locking or to implement application-layer validation. I chose a hybrid approach. Before any operation proceeds, the system calculates the aggregated balance and validates that the requested deduction will not result in a negative total. The deduction itself occurs within an ACID-compliant database transaction to ensure atomicity. This ensures that committed ledger entries are immediately visible to subsequent reads and prevents partial updates. I deliberately avoided heavy row-level locking strategies because the ledger design already reduces contention, and I wanted to preserve throughput as the system scales.

Another major concern was protecting against duplicate requests caused by network instability or client retries. Without safeguards, a single user action could result in multiple credit deductions. I decided to implement idempotency at the application layer rather than relying on infrastructure-level solutions. When a client includes an idempotency key, the system checks the ledger to determine whether a transaction associated with that key already exists for the organization. If it does, the request is treated as already completed and no additional deduction occurs. If not, the deduction proceeds and the key is permanently associated with that transaction. This binds idempotency directly to the financial record itself, which I believe is more robust than storing idempotency state in a separate cache that could expire or desynchronize from the ledger.

Background worker failures presented a subtle but critical edge case. The system deducts credits immediately when a request is accepted, then processes the actual task asynchronously. I considered delaying credit deduction until after successful job completion, but that approach would introduce the risk of users submitting many expensive jobs without sufficient credits, potentially overwhelming the system before validation occurs. Deducting at the API layer prevents overdrafts and enforces economic constraints immediately. However, this creates a risk if a background worker crashes after credits have already been deducted. To address this, I implemented a compensation mechanism. If a worker fails due to an exception or timeout, a new transaction is inserted into the ledger that refunds the exact deducted amount. This keeps the ledger internally consistent and ensures that organizations are never penalized for internal system failures. I chose a compensatory transaction instead of deleting or modifying the original deduction because immutability remains a core principle of the ledger. Every financial event, including corrections, is recorded as a separate entry.

Scalability was another significant consideration. In its current form, the system calculates the balance by aggregating all transactions for an organization on every request. This is acceptable at low to moderate scale, but it will eventually become the primary bottleneck as the ledger grows. I intentionally accepted this short-term inefficiency because it simplifies correctness during the early stage of development. Premature optimization would have complicated the system before its behavior was fully validated. For future scaling, I plan to introduce a hybrid caching strategy using Redis. The derived balance can be cached with a short time-to-live and invalidated or adjusted whenever a new transaction occurs. This preserves the ledger as the ultimate source of truth while dramatically reducing database load for read-heavy workloads. I also considered implementing materialized views or periodic snapshotting, where historical transactions are aggregated into starting balances to reduce the number of rows scanned during calculations. These optimizations would maintain correctness while improving performance at higher load.

Throughout the design process, I consistently prioritized financial integrity, auditability, and deterministic behavior over simplicity or immediate performance gains. In a system that manages user credits, correctness must be absolute. It is far easier to optimize performance than to repair lost trust or corrupted financial data. Every decision—from the append-only ledger to compensatory refunds and idempotency enforcement—reflects that philosophy. The system is built around the principle that financial state should always be reconstructable from immutable historical events. Performance enhancements are layered on top of that foundation, not embedded at the cost of clarity or safety.

In summary, the architecture deliberately favors immutability, atomicity, and explicit recovery mechanisms. Each tradeoff was evaluated in terms of risk, failure modes, and long-term maintainability. By centering the design around an append-only ledger and reinforcing it with idempotency guarantees and compensation logic, the system achieves strong correctness properties while remaining adaptable to future scaling improvements.
